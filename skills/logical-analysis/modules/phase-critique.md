# Phase 2: 다관점 에이전트 팀 비판 (Multi-Perspective Critique)

> **SOT**: 이 파일은 Phase C(project-planner) / Phase 2(logical-analysis)의 방법론 단일 진실 원천이다.
> **목적**: Phase 1의 구조를 복수의 독립된 관점으로 비판하고, 관점 간 충돌을 통해 단일 관점에서는 보이지 않는 약점을 드러낸다.

---

## 핵심 원칙

- Phase 1 결과를 방어하지 않는다
- 에이전트 간에도 상대의 비판을 방어하지 않고, 놓친 것을 찾는다

---

## 2-0. 에이전트 팀 구성

### 고정 에이전트 (모든 분석에 항상 참여)

**① 구조주의자 (Structuralist)**
- **렌즈**: 논리적 정합성과 의존 관계의 완전성
- **핵심 질문**: "이 구조에 빠진 연결, 숨겨진 의존, 순환 논증이 있는가?"
- **일반 분석용 검사 대상**:
  - Phase 0의 구성요소 분해가 MECE(상호 배타적·전체 포괄)인가?
  - Phase 1의 속성 간 의존 관계에 빠진 간선이 있는가?
  - 정의(2단계)와 전제(1단계) 사이에 순환이 있는가?
  - 충분성 검토(5단계)에서 "필요충분"으로 추정한 것이 실제로는 필요조건에 불과하지 않은가?

**② 회의론자 (Skeptic)**
- **렌즈**: 반례, 경계 조건, 붕괴 시나리오
- **핵심 질문**: "이 구조가 무너지는 경우는? 구조가 견고해 보이는 것은 단지 편리한 사례만 봤기 때문이 아닌가?"
- **일반 분석용 검사 대상**:
  - Phase 1의 9단계(반례)에서 발견하지 못한 반례
  - Phase 0의 피드백 루프가 발산/교착하는 시나리오
  - "정상 작동"을 전제한 분석이 장애 상황에서 어떻게 무너지는가
  - 구조주의자가 "정합적"이라 판단한 것이 실제로는 트리비얼하게 참인 것은 아닌가

**③ 실용주의자 (Pragmatist)**
- **렌즈**: 실제 구현/적용 시의 실현 가능성과 트레이드오프
- **핵심 질문**: "이론적으로 성립하지만 실제로는 불가능하거나 비용이 지나치게 높은 부분은 어디인가?"
- **일반 분석용 검사 대상**:
  - Phase 1에서 도출한 속성이 실제로 측정/구현 가능한가
  - 시스템의 이상적 작동 조건이 현실에서 얼마나 자주 충족되는가
  - 구조주의자와 회의론자의 비판이 실제로 중요한 문제인가, 아니면 이론적으로만 의미 있는 문제인가
  - 제안된 해결책의 비용 대비 효과가 현실적인가

### 동적 에이전트 (분석 대상에 맞춰 생성)

Phase 0의 결과를 바탕으로 1~3명의 추가 에이전트를 생성한다.

**생성 기준**:
- **(a)** 시스템의 핵심 긴장(tension)을 대표하는 관점
  - 예: AI 에이전틱 루프 → "안전성 옹호자" (자율성 vs 통제)
- **(b)** 시스템의 주요 이해관계자를 대표하는 관점
  - 예: AI 에이전틱 루프 → "최종 사용자 대리인" (시스템 vs 사용자 경험)
- **(c)** 인접 분야에서 유사 구조를 다룬 관점
  - 예: AI 에이전틱 루프 → "제어 이론가" (피드백 시스템의 안정성)

**각 동적 에이전트에 대해 명시할 것**:
- 이름과 렌즈 (한 문장)
- 왜 이 관점이 이 시스템 분석에 필요한가 (한 문장)
- 검사 대상 (3~5개)

### 적대적 포지션 할당

메인 오케스트레이터가 Phase 1 종합의 "핵심 주장" 산문에서 **개별 주장을 명시적으로 추출**하여 번호를 매긴다. 각 에이전트에게 추출된 주장 중 하나를 **반증 대상**으로 할당한다. 에이전트의 1차 임무는 할당된 주장이 틀렸음을 증명하는 것이며, 반증에 실패한 부분만 "유지 가능"으로 판정한다.

**할당 방식**:
- 핵심 주장의 수가 에이전트 수 이상이면: 1인 1주장 할당
- 핵심 주장의 수가 에이전트 수 미만이면: 가장 중요한 주장에 2명 할당 (서로 다른 렌즈로 반증 시도)
- 각 에이전트는 할당된 주장 외에도 자신의 렌즈로 추가 비판을 수행한다

이 할당은 LLM의 동의 편향(agreement bias)을 구조적으로 억제하기 위한 것이다. "약점을 찾아라"보다 "이것이 틀렸음을 증명하라"가 더 강한 비판을 유도한다.

### 증거 수집 규칙

Phase 2 에이전트는 비판 시 가능한 한 **외부 증거**를 수집하여 논거를 뒷받침해야 한다. 논리적 추론만으로는 비판이 "폐쇄 회로 자기 검증"에 그칠 수 있다.

**분석 대상이 코드/시스템인 경우**:
- Read, Grep, Glob, Bash(읽기 전용)로 실제 코드를 확인한다
- "이 구성요소가 단일 실패 지점이다" → 에러 핸들링 코드를 grep하여 검증
- "이 인터페이스가 병목이다" → 실제 호출 패턴을 코드에서 확인
- "이 전제가 성립한다" → 테스트 코드에서 검증 여부를 확인
- **반증 가능한 예측**을 만들고 도구로 검증하는 패턴을 권장한다:
  - [주장] → [예측: "이것이 참이면 코드에서 X를 발견할 것"] → [검증] → [결과]

**분석 대상이 개념/시스템인 경우**:
- WebSearch, WebFetch로 학술 문헌, 실제 사례, 통계 데이터를 검색한다
- "이 트레이드오프는 불가피하다" → 동시 달성 사례를 검색하여 확인
- "이 구조가 최적이다" → 대안 구조의 실제 구현 사례를 검색
- 검색 결과가 없거나 불충분한 경우, "외부 증거 부재"를 명시한다

**각 핵심 문제에 증거 유형을 표기한다**:
- `[코드 검증]`: 실제 코드에서 확인한 증거
- `[문헌 확인]`: 외부 문헌/데이터에서 확인한 증거
- `[논리적 추론]`: 외부 증거 없이 논리적으로 도출한 판단
- `[미검증]`: 검증을 시도했으나 확인 불가

**도구 사용 불가 시 fallback**: 서브에이전트에서 WebSearch/WebFetch가 사용 불가한 경우, 해당 증거를 `[미검증]`으로 태깅한다. Phase 2 R2에서 메인 오케스트레이터가 `[미검증]` 태그가 붙은 핵심 문제 중 외부 검증이 가능한 것을 선별하여 직접 WebSearch/WebFetch로 보완 검증한다. 보완 결과를 R2-g(잔여 불확실성)에 반영한다.

**팀 구성 제약**: 전체 에이전트 수 최소 4명, 최대 6명 (고정 3 + 동적 1~3). 개요 깊이에서는 동적 에이전트 없이 고정 3명만. 팀 구성 확정 후 확인 대기 없이 바로 Round 1로 진행한다.

## 2-1. Round 1 — 독립 비판

각 에이전트를 **별도 서브에이전트로 병렬 실행**한다. 각 에이전트는 독립된 컨텍스트에서 Phase 1의 결과를 비판하며, 다른 에이전트의 비판을 볼 수 없다.

각 서브에이전트에는 Phase 1 **종합**과 함께 **모든 구성요소의 key-findings 블록**(핵심 통찰, 취약한 전제, 역방향 함의, Phase 2 비판 대상)을 전달한다. 종합만 전달하면 축약 과정에서 반례·경계 조건의 상세가 소실되어 비판 범위가 제한된다.

### 독립성 보장

- 각 에이전트의 비판은 자신의 렌즈에만 근거해야 한다
- 서브에이전트로 실행하므로 다른 에이전트의 출력에 접근할 수 없다 (구조적 독립성)

### 프롬프트 템플릿

```
너는 {agent_name} 에이전트이다. Phase 1 분석 결과를 {lens} 관점에서 독립적으로 비판한다.

## 분석 맥락
- 분석 대상: {analysis_target_summary}
- 분석 목적: {analysis_purpose}
- 분석 깊이: {depth}

## Phase 1 결과 (비판 대상)

### Phase 1 종합
{phase_a_synthesis}

### 구성요소별 Key Findings
{all_key_findings}

{inspection_targets}

## 규칙

## 적대적 임무
너에게 할당된 반증 대상 주장: "{assigned_claim}"

- 반증 성공: "이 주장은 다음 이유로 기각되어야 한다: ..."
```

## 2-2. Round 2 — 판정 및 구조 수정

오케스트레이터가 직접 수행한다. Round 1의 독립 비판을 종합하여 시스템 수준의 최종 판정을 내린다. 특정 에이전트의 관점이 아니라, 비판의 결과를 객관적으로 정리하고, 그 결과를 바탕으로 Phase 1 핵심 주장의 존폐를 판정하는 **"판정관(Adjudicator)"** 역할이다. "합의/미합의" 판정은 Round 1의 독립 비판 간 내용 일치 여부로 판단한다.

> **실행 범위**: 분석 깊이가 "심층 (Deep)"일 때 R2-a ~ R2-g 전체를 작성한다.
> "개요 (Quick)"에서는 **R2-a(합의된 문제), R2-e(핵심 주장 판정), R2-g(잔여 불확실성)**만 작성한다.
> 생략된 섹션(R2-b, R2-c, R2-d, R2-f)의 핵심 내용이 있으면 R2-g에 간략히 언급한다.

각 하위 섹션은 **번호가 매겨진 항목 + 산문 설명**으로 작성한다. 불릿 나열로 끝내지 않고, 각 항목이 왜 중요한지, 토론에서 어떻게 도출되었는지를 문장으로 설명한다.

### R2-a. 합의된 문제 (Consensus Issues)

각 합의된 문제에 대해 다음을 문단으로 서술한다:
- 문제의 내용과 왜 합의에 이르렀는가
- 어떤 에이전트들이 동의했는가
- 심각도 판정 (치명적 / 중대 / 경미)과 그 근거

### R2-b. 미해결 쟁점 (Unresolved Disputes)

각 미해결 쟁점에 대해 다음을 문단으로 서술한다:
- 쟁점의 내용과 각 측의 핵심 논거 요약
- 왜 합의에 이르지 못했는가를 구체적으로 설명
- 해소를 위해 필요한 추가 정보 또는 분석

### R2-c. 분해 손실 점검

다음 질문들에 대해 각각 문단으로 답한다:
- Phase 0~1의 분해 과정에서 "빠진 것"이 있는가? 구성요소 단위로 분해할 때 사라지는 시스템 속성은 무엇인가?
- 에이전트 팀의 관점 조합으로도 포착하지 못한 것이 있는가? 팀 구성 자체가 만드는 맹점은 무엇인가?
- 분해 방식 자체가 달랐다면 어떤 것이 더 잘 보였을지를 구체적으로 서술한다

### R2-d. 구조 수정안

각 수정안에 대해 다음을 문단으로 서술한다:
- Phase 0 또는 Phase 1의 어느 부분에서 수정이 필요한가
- 수정 전/후를 대비하여 어떻게 달라지는가
- 수정이 다른 부분에 미치는 파급 효과

### R2-e. 핵심 주장 판정 ← 핵심 신규 섹션

Phase 1 종합의 핵심 주장 각각을 Round 1의 독립 비판 결과에 비추어 4가지 범주로 판정한다:

- **유지**: 비판을 견뎌낸 주장. 어떤 에이전트의 어떤 비판을 견뎌냈는지 근거를 1~2문장으로 명시한다. 복수 에이전트가 동일 주장을 비판한 경우, 가장 강력한 비판과 그것을 견딘 이유를 중심으로 서술한다.
- **수정**: 비판을 수용하여 변경된 주장. "무엇이 어떻게 바뀌었는가"를 첫 문장에서 명시하고, 어떤 에이전트의 비판이 수정을 촉발했는지를 1~2문장으로 서술한다. 수정이 Phase 3의 최종 답변에 미치는 영향도 언급한다.
- **추가**: 토론에서 새로 드러난 통찰. 어떤 에이전트의 어떤 라운드에서 도출되었는지를 1문장으로 명시하고, Phase 1에서 놓쳤던 이유를 1문장으로 설명한다.
- **폐기**: 비판으로 인해 더 이상 유효하지 않은 주장. 어떤 비판이 결정적이었는지를 1문장으로 서술하고, 이 주장에 기반한 Phase 1 잠정 답변 부분이 있으면 Phase 3에서 어떻게 대체해야 하는지를 명시한다.

각 항목을 산문 문단으로 서술한다. 불릿 나열로 끝내지 않는다.

### R2-f. 대안 구조 제시

- 같은 목적을 달성하는 다른 시스템 구조가 가능한지 서술한다
- 토론에서 드러난 근본적 한계를 해결하는 구조와 현재 구조 대비 트레이드오프를 설명한다

### R2-g. 잔여 불확실성 선언

각 불확실성에 대해 다음을 문장으로 서술한다:
- 결론 중 확신도가 낮은 것과 그 이유
- 추가 분석이 필요한 영역
- "이것이 참이면 이 결론이 뒤집힌다"는 조건을 구체적으로 명시한다

## 실행 안전장치

### 에이전트 실패 핸들링

**1단계 — 실패 감지 기준**:
- 에이전트가 응답하지 않거나 에러를 반환한 경우
- 출력에 "핵심 문제"와 "잠정 판정"이 모두 없는 경우 (형식 위반)
- 렌즈와 무관한 내용만 출력한 경우 (역할 이탈)

**2단계 — 1회 재시도**: 실패한 에이전트만 동일 프롬프트로 1회 재시도한다.

**3단계 — 재시도 실패 시**: 해당 에이전트를 제외하고 진행한다. R2-c(분해 손실 점검)에 실패한 에이전트의 렌즈가 커버하던 영역을 명시적으로 기록한다.

**4단계 — 과반 미만 성공 시**: Round 2는 성공한 Round 1 결과만으로 수행하되 "불완전 비판"임을 명시한다.
