# Phase 0~1: 시스템 위상 분석 + 구성요소 심층 분석

> **SOT**: 이 파일은 Phase A(spec-agent) / Phase 1(logical-analysis)의 방법론 단일 진실 원천이다. Phase 0(시스템 위상 분석)도 포함한다.
> **목적**: Phase 0에서 식별된 각 구성요소(또는 단일 개념 자체)를 지정된 깊이까지 해체한다.

---

## Phase 0: 시스템 위상 분석 (Topology)

> **목적**: 개별 구성요소를 파기 전에, 시스템의 전체 지형도를 먼저 그린다.
> **비유**: 도시를 분석하려면, 개별 건물을 조사하기 전에 먼저 도시 전체의 지도를 펼쳐야 한다.

### 적용 조건 — 복합 시스템 판별

다음을 **모두** 충족해야 "복합 시스템"으로 판별하여 이 Phase를 적용한다:
- 구성요소가 2개 이상이며 서로 상호작용한다
- **분석 목적상 내부 구조를 분리하여 분석할 실익이 있다** (목적 기반 판별)
- 다음 중 하나 이상에 해당한다:
  - 피드백 루프가 존재한다
  - "전체는 부분의 합과 다르다"는 직관이 있다
  - 시스템의 경계가 자명하지 않다

판별이 애매한 경우, **단일 개념으로 시작**한다. Phase 1 진행 중 복합 시스템임이 판명되면 이 Phase로 돌아간다.

### 0-1. 시스템 경계 설정

S의 "안"과 "밖"을 구분하는 기준을 설정한다:
- S가 전제하는 외부 환경(입력, 의존 대상)
- 경계 설정이 분석 결과에 미치는 영향을 문장으로 명시한다

### 0-2. 구성요소 식별

S를 이루는 주요 구성요소를 나열한다:
- 각 구성요소에 **서술적 이름**을 부여하고, 최초 나열 시 참조 번호를 괄호 안에 한 번만 표기할 수 있다 (예: "오케스트레이터(C₁)"). **이후 분석에서는 서술적 이름만 사용한다.**
- 각 구성요소가 "더 분해 가능한 하위 시스템"인지 "원자적(atomic)"인지 판별하고, 그 판별의 근거를 문장으로 설명한다
- 분해 가능하지만 여기서 멈추는 경우, 이유를 문장으로 명시한다

### 0-3. 인터페이스 매핑

구성요소 간에 무엇이 오가는지(데이터, 제어 신호, 상태, 피드백)를 서술한다:
- 각 인터페이스가 왜 존재하는지, 그것이 없으면 시스템이 어떻게 달라지는지를 설명한다
- 각 인터페이스의 방향성(단방향/양방향/조건부)과 그 이유를 문장으로 설명한다
- **Mermaid 다이어그램**으로 전체 구조를 보조적으로 시각화한다. 다이어그램만으로 관계를 전달하지 않는다

### 0-4. 피드백 루프 식별

구성요소들이 서로를 거쳐 원래 출발점으로 되돌아오는 순환 구조를 분석한다:
- 어떤 구성요소들이 어떤 순서로 연결되는지, 왜 그 순환이 발생하는지를 서술한다
- 각 순환이 변화를 증폭하는가(양성 피드백), 아니면 안정화하는가(음성 피드백)? 그 구분의 근거를 설명한다
- 순환이 한 바퀴 도는 데 얼마나 걸리며, 어디서 지연이 발생하는가?

### 0-5. 작동 순서와 생명주기

- 구성요소들의 실행 순서 (순차적 / 병렬 / 이벤트 기반 / 혼합)
- 시작 조건, 정상 상태, 종료 조건
- 시간에 따른 구성요소 역할 변화 여부

### 0-6. 창발 속성 후보

- 개별 구성요소에는 없지만 시스템 전체에서 나타나는 속성 후보
- 어떤 구성요소 조합에서 발생하는지 서술한다
- Phase 2(다관점 비판)에서 재검토할 것이므로 여기서는 후보 목록으로 충분하다

### Phase 0 출력물

이 Phase의 출력은 **0-1~0-6 단계의 전체 분석 결과**이다. Phase 1과 Phase 2에 입력으로 전달된다.

HITL 요약(구성요소 목록, 한 줄 설명, "원자적/하위 시스템" 판별 등)은 이 Phase의 별도 산출물이 아니라, 오케스트레이터가 분석 결과를 바탕으로 사용자에게 제시할 때 인라인으로 생성한다.

## 적용 규칙

- Phase 0에서 **원자적**으로 판별된 구성요소 → 1차(1~5단계) + 2차(6~9단계)를 1회 적용
- Phase 0에서 **하위 시스템**으로 판별된 구성요소 → Phase 0 재귀 적용 후, 하위 구성요소 각각에 1차+2차 적용
- "특정 영역 집중" 선택 시 → 해당 영역은 최대 깊이로, 나머지는 요약 수준

## 라운드 구조 (복합 시스템)

복합 시스템의 Phase 1은 **2라운드**로 실행한다. 단일 개념 분석에서는 9단계를 1회 실행하고, Round 2에서 교차검증 없이 바로 종합을 작성한다.

### Round 1: 병렬 구성요소 분석

선택된 각 구성요소를 **별도 서브에이전트로 병렬** 실행한다.

**서브에이전트 프롬프트에 포함할 핵심 지시:**
> 이 구성요소를 독립적으로 분석하되, Phase 0에서 식별된 다른 구성요소와의 인터페이스를 인식하라. 분석 중 다른 구성요소에 대한 함의를 발견하면 key-findings의 "역방향 함의"에 기록하라.
> 재귀 분해 시, 새로운 구조가 발견되지 않으면 "정보 이득 소진"을 선언하고 중단하라. 절대 상한은 5단계이다.

각 서브에이전트의 출력: 해당 구성요소의 9단계 분석 전문 + 출력 말미에 **key-findings 블록** (아래 템플릿 참조).

### Round 2: 중앙 교차검증

**모든 구성요소 분석 결과를 종합**하여 god's-eye view로 교차검증한다.

**수행 작업:**
1. **순방향 교차 참조 식별**: 구성요소 A의 출력/특성이 구성요소 B의 전제/작동에 영향을 미치는 관계를 모두 식별
2. **역방향 함의 검증**: 각 구성요소의 key-findings에 기록된 "역방향 함의"를 검토하고, 기록되지 않은 추가 역방향 함의도 탐색
3. **심각도 판정**: 발견된 교차 참조 각각에 대해:
   - **경미**: Phase 1 종합에서 언급으로 충분 (관계의 존재만 확인)
   - **중대**: 해당 구성요소의 분석 자체가 달라져야 함 (분석의 렌즈가 바뀜)
4. **Phase 1 종합 작성** (아래 "Phase 1 종합" 섹션 참조)

교차검증에서 중대한 교차 참조가 발견되면, 해당 구성요소의 분석을 보강하여 Phase 1 종합에 반영한다.

## 1차: 구축 (1~5단계)

각 단계는 최소 한 문단(2~3문장) 이상의 연결된 서술로 작성한다. 단계별 내용을 불릿 한 줄로 끝내지 않는다.

### 1. 전제

이 구성요소가 성립하려면 먼저 무엇이 참이어야 하는가?
- 시스템 맥락에서의 전제 포함 (예: "투자자 심리가 형성한 기대가 먼저 존재해야 수급 구조가 작동한다")
- 정의(2단계)를 먼저 시도하다가 전제를 발견하고 돌아오는 것도 자연스럽다

### 2. 정의

"이 구성요소가 존재한다/작동한다"는 정확히 무엇을 의미하는가?
- 시스템 내에서의 역할(function)과 독립적 정의(identity)를 구분한다

### 3. 원시 속성 도출

정의가 성립하려면 어떤 하위 조건/속성이 필요한가?
- 분석 목적과 깊이에 맞는 수준까지 분해한다
- 여기서 멈추는 경우 이유를 명시한다

### 4. 속성 간 의존 관계

도출된 속성들 사이의 전제/의존 관계를 서술문으로 설명한다. "A는 B에 의존한다"가 아니라 "왜 A가 B 없이는 성립할 수 없는가"를 문장으로 풀어 쓴다.
- Mermaid 다이어그램은 서술을 보충하는 용도로만 사용한다. 다이어그램만으로 관계를 전달하지 않는다

### 5. 충분성 검토

도출한 속성들이 모두 충족되면 이 구성요소가 반드시 성립하는가?
- 각 속성이 필요조건/충분조건/필요충분조건인지 **추정**한다. 확신도 수식어를 붙인다 (예: "필요조건으로 보인다", "충분조건일 가능성이 높다"). 형식 논리적 증명이 아닌 자연어 추론의 한계를 인식한다
- 미발견 속성 가능성을 명시한다

## 2차: 비판 (6~9단계)

1차의 결과를 비판적 대상으로 취급한다. 구축한 구조를 방어하려 하지 말고 적극적으로 약점을 찾는다.

각 단계는 논증 구조(주장 → 근거 → 예시 → 함의)를 갖춘 문단으로 작성한다. "이 속성에는 이런 문제가 있다"로 끝내지 말고, 왜 문제인지, 어떤 상황에서 드러나는지, 그것이 전체 구조에 어떤 영향을 미치는지까지 서술한다.

### 6. 속성의 스펙트럼

각 속성이 높음/중간/낮음/없음일 때 구성요소가 어떻게 달라지는가? 시스템 전체에 미치는 영향을 구체적 예시와 함께 서술한다.

### 7. 트레이드오프와 불가능성

동시에 달성할 수 없는 속성 쌍, 다른 구성요소와의 동시 최적화가 불가능한 것을 서술한다.

### 8. 경계

적용/비적용 영역의 경계, 속성 조합이 만드는 구조 전체의 붕괴 지점을 서술한다.

### 9. 반례

도출된 구조가 성립하지 않는 사례와 취약 지점을 명시한다. 반례가 구조 수정을 요구하면 수정된 구조를 제시한다.

## key-findings 블록 템플릿

각 구성요소 분석 결과의 **말미에** 다음 형식으로 작성한다:

```markdown
---
## Key Findings: {구성요소 서술적 이름}

### 핵심 통찰
[이 구성요소에 대해 분석이 도달한 핵심 결론. 1~2문장]

### 취약한 전제
[이 구성요소의 분석이 의존하는 전제 중, 다른 구성요소 분석에서 뒤집힐 수 있는 것]
- "{전제 내용}" — 거짓이면 {어떤 결론이 무효화되는지}

### 역방향 함의
[이 구성요소 분석에서 발견된 것 중, 다른 구성요소의 분석을 수정해야 할 가능성이 있는 것]
- {대상 구성요소명}: {수정이 필요한 내용과 이유}

### 핵심 반례와 경계 조건
[8단계(경계)에서 도출된 가장 중요한 붕괴 지점 1개 + 9단계(반례)에서 도출된 가장 심각한 반례 1개를 각 1~2문장으로 서술]

### Phase 2 비판 대상
[이 구성요소에서 Phase 2 에이전트가 비판해야 할 핵심 지점. 1~2개]
```

## 재귀 깊이 제어

- 분석 중인 구성요소가 상위 시스템에서 어떤 위치에 있는지를 자연어 전환 문장으로 설명한다
- **기본 깊이는 분석 깊이 설정에 따라 다르다**: 심층 (Deep) = 기본 2 / 최대 4, 개요 (Quick) = 재귀 분해 없음(1단계)
- 기본 깊이 이후 각 수준에서, 새로운 속성·의존관계가 발견되지 않으면 "정보 이득 소진"을 선언하고 중단한다. 중단 시 그 판단을 1~2문장으로 명시한다
- **절대 상한**: 심층 (Deep)은 4단계. 상한에 도달하면 무조건 중단한다

## Phase 1 종합

Round 2(중앙 교차검증)에서 작성한다. 모든 구성요소 분석 결과를 확보한 상태에서 작성하므로, 개별 서브에이전트가 발견하지 못한 교차 구성요소 패턴을 여기서 포착한다.

1차 구축과 2차 비판을 거친 **잠정적 결론**을 산문으로 서술한다. 이 종합은 Phase 2의 비판 대상이 되며, Phase 3에서 수정될 것을 전제한다.

**서술할 내용**:
- **핵심 주장**: 이 시스템/개념에 대해 분석이 도달한 잠정적 결론. 2~3문장의 산문으로, 구축(1~5단계)과 비판(6~9단계)을 모두 반영한다.
- **구성요소 간 관계의 전체상**: 개별 구성요소 분석에서 드러난 교차 참조와 인과 사슬을 종합한다. 순방향(A의 출력이 B의 전제) 및 역방향(B의 분석이 A의 전제를 뒤집음) 관계를 모두 포함한다.
- **구축-비판 긴장**: 1차에서 구축한 구조와 2차 비판 사이의 주요 긴장 지점을 명시한다. 해소된 것과 미해소된 것을 구분한다.
- **잠정 답변**: 분석 목적에 대한 1차 답변을 제시한다. 이 답변은 Phase 2 비판을 아직 거치지 않은 상태임을 명시한다.
- **주요 반례와 경계 조건**: 각 구성요소 분석의 8단계(경계)와 9단계(반례)에서 도출된 핵심 발견을 수집한다. 구성요소별로 가장 심각한 반례 1~2개와 가장 중요한 경계 조건 1~2개를 명시한다.

## 실행 안전장치

### 서브에이전트 실패 핸들링

**1단계 — 실패 감지 기준**:
- 서브에이전트가 응답하지 않거나 에러를 반환한 경우
- 9단계(1차 구축 5단계 + 2차 비판 4단계) 중 일부만 출력한 경우 (미완료)
- key-findings 블록이 출력에 포함되지 않은 경우 (형식 위반)

**2단계 — 1회 재시도**: 실패한 서브에이전트만 동일 프롬프트로 1회 재시도한다.

**3단계 — 재시도 실패 시**: 해당 구성요소를 제외하고 진행한다. Phase 1 종합에 "구성요소 '{이름}'의 분석이 불완전함 — 서브에이전트 실패로 제외됨"을 명시한다.

**4단계 — 과반 미만 성공 시**: 전체 구성요소 중 과반 미만만 성공했을 경우, Phase 1 종합에 "⚠️ 불완전 분석 — 전체 구성요소의 과반 미만만 분석 완료"를 명시하고, Phase 2 에이전트에게도 이 사실을 전달한다.

## 백트래킹 시그널

분석 중 "단일 개념"이 실제로 "복합 시스템"임이 판명되면, 서브에이전트는 다음 절차를 따른다:

1. **즉시 분석을 중단**한다. 현재까지의 부분 결과는 폐기한다.
2. **반환값 최상단에 백트래킹 마커를 포함**한다:
   ```
   [BACKTRACK: complex-system-detected]
   근거: {왜 복합 시스템으로 판단했는지 1~2문장}
   ```
3. 마커 이후에 추가 분석 내용을 출력하지 않는다.

**메인 오케스트레이터 처리**: 서브에이전트 반환값의 첫 100자 이내에 `[BACKTRACK:` 문자열이 존재하면 백트래킹으로 판정한다. 판정 시 Phase 0을 실행하여 시스템 위상 분석부터 재시작한다.
