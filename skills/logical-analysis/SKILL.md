# 논리적 해체 분석

단일 개념부터 복합 시스템까지, 논리적 근거·특성·구성요소·연관관계를 체계적으로 해체하는 분석 스킬. 다관점 에이전트 팀의 토론형 비판 구조를 포함한다.

## 워크플로우 개요

1M 컨텍스트 윈도우를 활용하여 메인 오케스트레이터가 Phase 0~3을 직접 실행하고, 서브에이전트는 독립성·병렬성이 필요한 곳(Phase 1 R1, Phase 2 R1/R2)에서만 사용한다.

```
사전 질문 → 맥락 확보
    ↓
단일/복합 판별
    ↓
Phase 0 (메인 직접, 복합만)
    ↓
  ★ HITL: 구성요소 선택 + 깊이 지정
    ↓
Phase 1 (메인 조율)
    → R1: 구성요소별 서브에이전트 병렬 spawn
    → R2: 메인이 직접 교차검증
    ↓
Phase 2 (메인 조율)
    → 팀 구성: 메인 직접
    → R1: 비판 에이전트 서브에이전트 병렬 spawn
    → R2: 메인이 직접 판정
    ↓
Phase 3 (메인 직접)
    ↓
최종 파일 저장
```

**단일 개념** (예: 신뢰, 자유, 창의성): Phase 0 건너뛰고 Phase 1 직행
**복합 시스템** (예: AI 에이전틱 루프, 마이크로서비스): Phase 0부터 시작

판별 기준 — 다음을 **모두** 충족해야 "복합 시스템":
- 구성요소가 2개 이상이며 서로 상호작용한다
- **분석 목적상 내부 구조를 분리하여 분석할 실익이 있다** (목적 기반 판별)
- 다음 중 하나 이상에 해당한다:
  - 피드백 루프가 존재한다
  - "전체는 부분의 합과 다르다"는 직관이 있다
  - 시스템의 경계가 자명하지 않다

판별이 애매한 경우, **단일 개념으로 시작**한다. Phase 1 R1 서브에이전트가 분석 중 "단일 개념"이 실제로 "복합 시스템"임을 판명하면 반환값에 백트래킹 신호를 포함하고 분석을 중단한다. 메인 오케스트레이터는 이를 감지하고 Phase 0을 실행한다.

## 실행 방식

### 실행 계층

분석은 2단계 계층으로 실행한다:

- **깊이 0 (메인 오케스트레이터)**: Phase 0~3 직접 실행 + 서브에이전트 조율
- **깊이 1 (서브에이전트)**: Phase 1 R1 구성요소 분석, Phase 2 R1 비판

모든 서브에이전트(깊이 1)에 본 문서의 "서술 규칙" 섹션을 전달한다.

HITL(Human-in-the-Loop)은 사전 질문과 Phase 0 완료 후 2회 발생한다.

### 서브에이전트 프롬프트 구성

서브에이전트에는 해당 라운드 수행에 필요한 섹션만 전달한다. 모듈 파일 전체를 통째로 전달하지 않는다.

**Phase 1 Round 1 서브에이전트**:
- `${MODULES}/phase-analysis.md`의 "1차: 구축" + "2차: 비판" + "key-findings 블록 템플릿" + "재귀 깊이 제어" 섹션
- 본 문서의 "서술 규칙" 섹션
- Phase 0 결과 요약 + 분석 맥락

**Phase 2 Round 1 비판 에이전트**:
- 해당 에이전트의 렌즈 정의 + 검사 대상 (phase-critique.md의 에이전트 정의에서 추출)
- Round 1 출력 형식 (phase-critique.md의 "Round 1 출력 형식" 섹션)
- 증거 수집 규칙 (phase-critique.md의 "증거 수집 규칙" 섹션)
- 적대적 임무 (할당된 반증 대상 주장)
- Phase 1 종합 + 전체 key-findings

**Phase 2 Round 1 비판 에이전트에 전달하지 않는 것**:
- phase-critique.md의 Round 2 구조, 에이전트 실패 핸들링, 팀 구성 규칙

### Phase 0 후 HITL 체크포인트

메인이 Phase 0 결과를 바탕으로 구성요소 목록을 직접 요약하여 사용자에게 제시한다.

사용자에게 다음을 확인한다:

1. **구성요소 선택**: "식별된 구성요소 중 어떤 것을 깊이 분석할까요?" — 불필요한 구성요소를 사전에 제거하여 출력 분량을 제어한다
2. **구성요소별 깊이**: 선택된 구성요소 각각에 대해 "심층 (Deep) / 개요 (Quick)" 중 택일. 기본값은 사전 질문에서 지정한 깊이
3. 사용자가 "전부 해줘", "알아서 해줘" 등 명시적 위임 표현을 사용하면 전체 구성요소를 사전 질문 깊이로 진행

단일 개념 분석(Phase 0 건너뜀)에서는 이 HITL을 생략하고 바로 Phase 1로 진행한다.

### 규모 안전장치

구성요소 수 × 깊이 조합이 다음 임계값을 초과하면 사용자에게 경고한다:
- 구성요소 5개 이상 × 심층(Deep): "⚠️ 구성요소 {N}개를 심층 분석하면 서브에이전트 {N+에이전트수}회 호출, 결과 파일 100+ 페이지가 예상됩니다. 축소하거나 개요(Quick)로 변경하시겠습니까?"
- 구성요소 8개 이상 (모든 깊이): "⚠️ 구성요소 {N}개는 교차검증 복잡도가 O(n²)입니다. 핵심 5개 이하로 축소를 권장합니다."

사용자가 그래도 진행하겠다고 하면 진행한다. 이 경고는 정보 제공용이며 차단이 아니다.

### 깊이↔실행 매트릭스

분해 깊이에 따라 각 Phase의 실행 방식이 달라진다.

| 분해 깊이 | Phase 1 | Phase 2 동적 에이전트 | Phase 3 |
|----------|---------|-------------------|---------|
| 심층 (Deep) | 적응적 깊이 (기본 2, 최대 4), 구축+비판 | 최대 2 (총 5명) | 전체 |
| 개요 (Quick) | 구축만 (비판 생략) | 0 (총 3명, 고정만) | 간략 |

## 진행 규칙

> **서술 및 서사 규칙**: 본 문서의 "서술 규칙" 섹션 참조. 모든 서브에이전트에 전달한다.

이하는 메인 오케스트레이터의 운영 규칙이다:

1. **컨텍스트 내 핸드오프**: 각 Phase의 출력은 메인 컨텍스트에 유지된다. Phase 1 R1과 Phase 2 R1만 서브에이전트(Agent 도구)로 실행한다.
2. **기본값 명시**: 사용자가 "모르겠다" 또는 "알아서 해줘"라고 하면 합리적 기본값을 사용하고, 어떤 기본값을 선택했는지 명시한다.
3. **Mermaid 검증**: 분석 결과를 파일로 저장하기 전에, 모든 Mermaid 다이어그램의 syntax를 `mmdc`로 검증한다. `mmdc` 미설치 시 검증을 건너뛰고 `⚠️ mmdc 미설치 — Mermaid 검증 생략됨` 경고를 출력한 후 파일 저장을 진행한다. (상세 절차는 아래 "분석 결과 저장" 섹션 참조)

## 사전 질문

분석을 시작하기 전에 **핵심 3개 질문**으로 맥락을 확보한다. 나머지는 합리적 기본값을 자동 적용한다.

### 사용자에게 묻는 질문 (3개)

**(a) 분석 목적**: 왜 이 시스템을 해체하려 하는가?
- 기본값: **학술적 이해** 또는 **구현을 위한 사전지식 조사**
- 사용자가 별도로 명시하지 않으면 위 두 가지 중 주제에 적합한 쪽을 자동 선택하고 명시한다
- "구현을 위한 사전지식 조사"로 판단되면 **무엇을 만들려는지**를 추가로 확인한다. 사용자의 요청에 구현 대상이 이미 언급되어 있으면 별도로 묻지 않고 그것을 반영한다.
- 그 외 목적(아키텍처 설계 / 설계 결함 탐색 / 대안 비교 등)은 주제에 따라 합리적인 값을 제안한다

**(b) 분해 깊이**: 어떤 수준까지 분해하길 원하는가?
- 심층 (Deep) / 개요 (Quick)
- 기본값: **심층 (Deep)** (충분한 깊이의 분석을 기본 제공)

**(c) 관심 초점**: 시스템 전체를 균일하게 볼 것인가, 특정 영역에 집중할 것인가?
- 균일 분석 / 특정 영역 집중 — 어디?
- 기본값: **균일 분석** (특정 영역 집중이 필요한 경우 사용자가 명시)

### 자동 적용되는 기본값

다음 항목은 사용자에게 묻지 않고 합리적 기본값을 적용한다. 적용한 기본값은 Phase 0 시작 시 명시한다.

| 항목 | 기본값 | 근거 |
|------|--------|------|
| **기존 전제** | 없음 (백지 상태) | 전제가 있는 사용자는 요청 시 자연스럽게 언급한다. 없는 사용자에게 묻는 것은 답할 수 없는 질문이다. |
| **시스템 경계** | 분석 목적에 따라 자율 설정 | Phase 0-1에서 경계를 명시하고, HITL 체크포인트에서 사용자가 수정할 수 있다. 사전에 묻는 것보다 Phase 0 결과를 보고 판단하는 것이 더 정확하다. |

사전 질문의 응답을 반영하여 이후 분석의 방향과 깊이를 조정한다.

## Phase 0: 시스템 위상 분석

> 방법론: `${MODULES}/phase-analysis.md`의 "Phase 0: 시스템 위상 분석" 섹션 참조
> 목적: 개별 구성요소를 파기 전에, 시스템의 전체 지형도를 먼저 그린다.

메인이 직접 실행한다. `${MODULES}/phase-analysis.md`의 "Phase 0: 시스템 위상 분석" 섹션의 방법론을 따른다. Phase 0 완료 후 구성요소 목록을 사용자에게 요약 제시하고 HITL 체크포인트를 진행한다.

단일 개념 분석 시 이 Phase를 건너뛴다.

## Phase 1: 구성요소 심층 분석

> 방법론: `${MODULES}/phase-analysis.md` 참조
> 목적: Phase 0에서 식별된 각 구성요소(또는 단일 개념 자체)를 사전 질문 (b)에서 지정한 깊이까지 해체한다.

메인이 2개 라운드를 직접 조율한다.

### 라운드 실행 구조 (복합 시스템)

복합 시스템의 Phase 1은 **2라운드**로 실행한다. 단일 개념 분석에서는 서브에이전트 1개로 전체 9단계를 실행하고, Round 2에서 교차검증 없이 바로 종합을 작성한다.

- **Round 1 (병렬)**: 메인이 구성요소별 서브에이전트를 **단일 메시지에서 병렬 spawn**한다. 각 서브에이전트에 Phase 0 결과 요약 + context + 본 문서의 "서술 규칙" 섹션 + `${MODULES}/phase-analysis.md`를 전달한다. 서브에이전트의 반환값이 메인 컨텍스트에 직접 들어온다.
- **Round 2 (메인 직접)**: 메인이 직접 수행한다. 모든 R1 결과가 이미 컨텍스트에 있으므로 파일 로드 없이 교차검증 + 종합을 작성한다. 교차검증에서 중대한 교차 참조가 발견되면, 해당 구성요소의 분석을 보강하여 Phase 1 종합에 반영한다.

### 백트래킹 핸들링

서브에이전트가 반환값에 백트래킹 신호를 포함하면, 메인이 이를 감지하고 Phase 0을 실행한다.

---

## Phase 2: 다관점 에이전트 팀 비판

> 방법론: `${MODULES}/phase-critique.md` 참조
> 목적: Phase 0~1의 구조를 복수의 독립된 관점으로 비판하고, 관점 간 충돌을 통해 단일 관점에서는 보이지 않는 약점을 드러낸다.

메인이 직접 조율한다.

### 실행 구조

- **팀 구성 (메인 직접)**: 메인이 직접 수행한다. Phase 1 종합을 바탕으로 `${MODULES}/phase-critique.md`의 방법론에 따라 에이전트 팀을 구성한다.
- **Round 1 (병렬)**: 메인이 비판 에이전트별 서브에이전트를 **단일 메시지에서 병렬 spawn**한다. 각 서브에이전트에 본 문서의 "서술 규칙" 섹션 + `${MODULES}/phase-critique.md`를 전달한다. **독립성 보장**: 서브에이전트이므로 다른 에이전트의 출력에 접근 불가.
- **Round 2 (메인 직접)**: 메인이 직접 수행한다. 모든 R1 결과가 컨텍스트에 있으므로 바로 판정관(Adjudicator) 역할로 종합을 작성한다.

---

## Phase 3: 최종 종합

> 방법론: `${MODULES}/phase-synthesis.md` 참조
> 목적: Phase 2에서 판정된 핵심 주장과 미해결 쟁점을 바탕으로, 분석 목적에 대한 최종 답변을 산출하고 분석 전체의 확신도를 평가한다.

메인이 직접 실행한다. `${MODULES}/phase-synthesis.md`의 방법론을 따른다.

Phase 3는 분석의 **최종 산출물**이다. "3-2. 분석 목적에 대한 최종 답변"의 내용만 YAML frontmatter 바로 뒤에 TL;DR 문단으로 복사하여, 독자가 결론을 서두에서 즉시 확인할 수 있게 한다. 3-2가 6문단을 초과하면, (a) 직접 답변과 (d) 목적별 핵심 산출물 문단만 TL;DR로 복사하고 나머지는 "상세는 Phase 3의 3-2 참조"로 안내한다.

## 서술 규칙

> 모든 Phase의 모든 실행 주체 (메인 오케스트레이터, 서브에이전트)에 적용.

1. **관계와 인과는 문장으로 풀어 쓴다** — 화살표 체인(→, ↑, ↓), 수식 형태의 압축 표현을 산문 본문에서 사용하지 않는다. 원인과 결과를 자연스러운 서술문으로 연결한다.
2. **구성요소는 서술적 이름으로 참조한다** — 코드명(C₁, P₂) 단독 참조 금지. 최초 소개 시 "투자자 심리(C₃)"처럼 병기하되, 이후 서술적 이름만 사용한다.
3. **각 단계는 문단 서술이 기본이다** — 불릿 한 줄로 단계 전체를 처리하지 않는다. 최소 2~3문장의 연결된 서술 후, 필요 시 보조적으로 표나 목록을 붙인다. 정보 유형에 따라 최적 표현 형식을 선택한다.
4. **접속어와 전환 표현을 적극 사용한다** — "이것은 ~때문이다", "따라서", "반대로" 등으로 논리적 연결을 드러낸다.
5. **다이어그램은 Mermaid만 사용한다** — ASCII 다이어그램 금지. 다이어그램은 산문 설명을 보충하는 역할이며, 다이어그램만으로 관계를 전달하지 않는다.

### 금지되는 서술 패턴

산문 본문에서 사용하지 않는다. Mermaid 다이어그램 코드 블록 내부에서만 기호 사용이 허용된다.

| 금지 패턴 | 예시 (X) | 대체 방식 (O) |
|---|---|---|
| 코드명 단독 참조 | `C₃의 P₂가 높으면` | "투자자 심리의 군집 강도가 높으면" |
| 불릿 전용 단계 서술 | `- 순차적 실행 모델` (한 줄로 단계 전체를 처리) | 최소 한 문단(2~3문장)의 연결된 서술 |
| 속성 코드 연산 | `P₁ + P₂ + P₄는 필요충분` | "깊이, 스프레드, 회복력이 모두 갖춰지면 건전하다고 추정할 수 있다" |

## 분석 결과 저장

Phase 3까지 모든 분석이 완료되면, 결과를 마크다운 파일로 저장한다.

### 저장 위치 및 파일명
- **경로**: `${OUTPUT_DIR}/{주제}-analysis-{YYYY-MM-DD}.md`
  - `${OUTPUT_DIR}` 디렉토리가 없으면 생성한다
  - `{주제}`: 분석 대상의 영문 kebab-case 요약 (예: `agentic-loop`, `trust`, `microservices`)
  - `{YYYY-MM-DD}`: 분석 수행 날짜
  - 예: `${OUTPUT_DIR}/agentic-loop-analysis-2026-03-04.md`
- 동일 파일이 이미 존재하면 `-v2`, `-v3` 등 버전 접미사를 붙인다

### 대용량 출력 분할 저장

최종 파일의 예상 크기가 50,000 토큰(약 200페이지)을 초과할 경우, 단일 파일 대신 디렉토리 구조로 분할 저장한다:

```
${OUTPUT_DIR}/{주제}-analysis-{YYYY-MM-DD}/
├── 00-overview.md    ← TL;DR + YAML frontmatter
├── 01-phase0.md      ← Phase 0 결과
├── 02-phase1.md      ← Phase 1 결과 (구성요소별 분석 + 종합)
├── 03-phase2.md      ← Phase 2 결과 (에이전트 비판 + 판정)
├── 04-phase3.md      ← Phase 3 결과 (최종 종합)
└── 05-limitations.md ← 이 분석의 한계
```

00-overview.md에 나머지 파일에 대한 링크를 포함한다.

### 최종 파일 작성

메인이 컨텍스트 내 Phase 0~3 결과를 종합하여 최종 분석 파일을 Write 도구로 저장한다. 파일 구조는 다음과 같다:

  1. YAML frontmatter
  2. **TL;DR** — Phase 3의 "3-2. 분석 목적에 대한 최종 답변" 내용을 그대로 복사한 문단. `## TL;DR` 헤더를 사용한다. 3-2가 6문단을 초과하면, (a) 직접 답변과 (d) 목적별 핵심 산출물 문단만 복사하고 "상세는 Phase 3의 3-2 참조"로 안내한다.
  3. Phase 0 결과
  4. Phase 1 결과 (구성요소별 분석 + 종합)
  5. Phase 2 결과 (에이전트 비판 + 판정)
  6. Phase 3 결과
  7. 이 분석의 한계 (본 문서의 "이 분석의 알려진 한계" 섹션 참조)

파일 최상단에 YAML frontmatter를 추가한다:
  ```yaml
  ---
  title: "논리적 해체 분석: {주제}"
  date: {YYYY-MM-DD}
  type: logical-analysis
  depth: {심층 | 개요}
  focus: {균일 | 집중 영역명}
  ---
  ```

### Mermaid 다이어그램 검증

파일 저장 전 모든 Mermaid 블록을 `mmdc`로 검증한다 (최대 3회 재시도). `mmdc` 미설치 시 `⚠️ mmdc 미설치 — Mermaid 검증 생략됨` 경고를 출력하고 저장을 진행한다.

## 이 분석의 알려진 한계

이 방법론의 구조적 한계를 인식하고 분석 결과에 반영한다:

1. **충분 조건의 완전한 확인은 원리적으로 불가능** — 모든 귀납적 분석의 본질적 한계. 충분성 검토(5단계)의 판정은 추정이지 증명이 아니다.
2. **구축과 비판의 진정한 분리는 불완전** — Phase 분리가 이를 부분적으로 완화하지만, 같은 LLM이 수행하는 근본적 한계가 있음. Phase 1의 프레이밍이 Phase 2 비판의 범위를 제약할 수 있다. Phase 2의 적대적 포지션 할당이 이 한계를 부분적으로 완화한다: 특정 주장의 반증을 명시적 임무로 부여하여 동의 편향을 억제한다.
3. **분해 방식의 비유일성** — 다른 구분 방식은 다른 인사이트를 줌. "해체"라는 방법론 자체가 환원주의적 전제를 내포하며, 본질적으로 통합적인 현상(의식, 미적 경험, 창발)에는 불리할 수 있다.
4. **에이전트 독립성의 부분적 한계** — Agent tool 병렬 실행으로 구조적 독립성을 보장하지만, 같은 LLM이 역할을 수행하므로 학습된 편향은 잔존한다. 2단계 중첩(메인 → 서브에이전트)으로 지시 전달 충실도 저하는 제한적이나, 1단계 위임에서도 역할 지시가 희석될 가능성은 남아 있다.
5. **LLM 사전 지식과 분석적 통찰의 구분 불가** — LLM이 이미 알고 있는 지식을 프레임워크에 맞춰 재포장할 수 있으며, 이것이 "새로운 분석적 통찰"과 구분되지 않을 수 있다.
6. **토론 수렴의 비보장** — 2라운드 구조가 반드시 합의로 이어지지 않는다. 미해결 쟁점은 솔직하게 "미해결"로 남기는 것이 거짓 합의보다 낫다.
