# Phase C 에이전트 분리 실행 프로토콜

> **적용 대상**: §VALIDATE Phase C
> **SOT**: `${LA_MODULES}/phase-critique.md`

Phase C(다관점 검증)의 Round 1(독립 비판)을 Agent tool로 병렬 실행하고, Round 2(판정)를 메인 에이전트가 판정관(Adjudicator) 역할로 직접 수행하는 프로토콜을 정의한다.

## §0 SOT 참조

다음 항목은 `${LA_MODULES}/phase-critique.md`를 SOT로 사용한다. 해당 파일을 읽어 적용한다:
- 에이전트 팀 구성 (고정 3명 + 동적 생성 기준 + 팀 크기 제약)
- Round 1 프롬프트 템플릿 + 출력 형식
- Round 2 판정관(Adjudicator) 역할 + 7섹션(R2-a~R2-g)
- 금지 패턴
- 에이전트 실패 핸들링 (1회 재시도 → 제외 → 과반 미만 시 불완전 비판 명시)

## §2 코드 검증용 검사 대상 (§VALIDATE) — project-planner 전용

**구조주의자**:
- R-ID → 서브태스크 → Oracle → 코드 변경의 정합성 체인에 누락된 연결이 있는가
- SPEC-TEST 매트릭스의 커버리지가 R-ID를 완전히 포괄하는가
- 서브태스크 간 의존 관계가 실행 순서에 올바르게 반영되었는가
- 코드 변경이 PLAN.md의 Action 명세와 일치하는가

**회의론자**:
- Oracle이 PASS를 반환하지만 R-ID가 실제로는 미달성인 시나리오 (false positive)
- 테스트가 커버하지 못하는 엣지 케이스
- 구현이 의도와 다르게 동작할 수 있는 조건 (입력 범위, 동시성, 에러 경로)
- "정상 경로에서만 테스트됨"의 위험

**실용주의자**:
- 프로덕션 환경에서의 실현 가능성 (부하, 타임아웃, 리소스 제약)
- 운영 리스크 (모니터링, 롤백 가능성, 의존 서비스 장애)
- 회귀 리스크 (기존 기능에 대한 의도치 않은 영향)
- 배포 후 첫 일주일에 발생할 수 있는 현실적 문제
